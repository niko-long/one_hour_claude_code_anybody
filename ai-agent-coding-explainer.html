01 大模型不是数据库

首先纠正一个常见的误解：大语言模型（LLM）不是在一个数据库里查找答案。它是通过大量文本训练出来的一组数学参数，在接收到输入后，一个 token 一个 token 地"预测"下一个最合理的内容。

这意味着几件事：

  • 它没有"记忆"——每次对话默认是独立的
  • 它不"知道"你之前说了什么，除非你把之前的内容再次发给它
  • 它的输出质量高度依赖你输入的内容（Garbage in, garbage out）
类比： 想象你在和一个极其聪明但失忆的人对话。每次你开口，他都需要你先把之前聊过的内容全部复述一遍，然后才能接上。

02 上下文窗口（Context Window）

上下文窗口是大模型在一次交互中能"看到"的全部文本范围，以 token 为单位计量（1 个中文字 ≈ 1-2 个 token，1 个英文单词 ≈ 1 个 token）。

上下文窗口的构成：系统指令 + 对话历史 + 当前输入 + AI 回复空间

关键点：对话历史和 AI 回复共享同一个窗口。前面塞得越多，留给回复的空间就越少，AI 的质量也越容易下降。

03 每轮对话到底发生了什么

这是很多人不了解的核心机制：每一轮对话，客户端都会把之前的全部历史重新发送给模型。 模型本身没有记忆，它看到的永远是"一整段文本"。

每轮对话的实际传输：前面所有轮次的内容被完整重发

成本影响： 这就是为什么长对话越来越贵。第 1 轮你发了 500 tokens，第 2 轮变成了 1500，第 3 轮 3000... 按 token 计费的 API 调用中，这个增长是线性累加的。订阅制用户虽然不直接感受到费用，但会遇到限额（rate limit）和响应速度变慢。

04 为什么要控制输入长度

理解了上下文窗口的机制后，控制输入长度的原因就很清晰了：

1. 噪音降低准确率

塞进去的无关信息越多，模型需要从中筛选有用部分的难度就越大。这就像在嘈杂的房间里试图听清一个人说话。

❌ 信息过载

把整个项目的 50 个文件全部贴进去，然后问"帮我改一下登录页"

→ 模型需要从大量无关代码中定位目标，容易遗漏或误改

✅ 精准输入

只贴登录相关的 3 个文件，说清楚要改哪里、改成什么样

→ 模型专注于小范围代码，准确率高

2. "迷失在中间"现象

研究表明，当输入文本很长时，模型对开头和结尾的内容关注度最高，中间部分容易被"遗忘"。这被称为 Lost in the Middle 现象。

输入过长时，中间部分的信息容易被模型忽略

3. 输出质量下降

当上下文窗口被塞满时，留给 AI 组织回复的"思考空间"变少。就像让你在一张已经写满字的纸上再写一篇文章。

05 为什么一个窗口只做一件事

了解了以上机制后，"一个对话窗口只关注一个项目/一个主题"的建议就不是随便说的了。

主题隔离 = 上下文纯净 = 输出质量高

核心原因： 由于每轮对话都会把前面全部历史重发，混在一起的无关内容会持续占用空间并干扰模型判断。开一个新窗口的成本为零，但混用窗口的代价是输出质量下降和上下文浪费。

06 Agent 循环

传统的 AI 对话是"你问一句，我答一句"。而 Agent 模式（如 Claude Code）是"你说目标，我自己想办法完成"。

Agent 内部是一个循环：接收任务 → 思考该怎么做 → 选择工具 → 执行 → 观察结果 → 判断是否完成 → 如果没完成，继续循环。

Agent 循环：Think → Act → Observe → 判断 → 循环或结束

和普通对话的区别： 普通对话是一问一答。Agent 模式下，你说"帮我写一个登录模块并测试"，Claude Code 会自动执行多轮操作 — 先读现有代码、规划结构、写代码、运行测试、分析错误、修复、再测试 — 可能跑 10+ 轮循环才返回最终结果。

Agent 可用的工具（以 Claude Code 为例）

工具作用
Read读取文件内容
Write创建或修改文件
Bash执行终端命令（运行代码、安装依赖等）
Search在代码库中搜索关键词
WebSearch联网搜索信息
MCP Tools调用外部服务（GitHub、数据库等）

07 Memory、Project 与 Instructions

既然模型本身没有记忆，平台层面提供了几种机制来弥补这个问题。以 Claude 为例：

Memory（记忆）

平台会从你的历史对话中提取关键信息（偏好、背景、项目上下文），在后续会话中自动注入到系统提示词里。

Memory 从历史对话中提取，注入到新会话的上下文中

注意： Memory 不等于"AI 记住了一切"。它是从对话中提取的摘要性信息，有选择性地注入。而且它也占用上下文窗口的空间。

Project（项目空间）

Project 是一个隔离的工作空间，可以关联特定的文件、Instructions 和知识库。同一个项目下的对话共享上下文设定，不同项目之间完全隔离。

概念作用生效范围
Memory跨会话的用户偏好和背景信息全局（除非在 Project 内有单独的 Memory）
Project隔离的工作空间，关联文件和设定Project 内的所有对话
Instructions自定义指令，规定 AI 的行为方式全局或 Project 级别

Instructions / Custom Instructions（自定义指令）

用户可以编写持久的指令来控制 AI 的行为。例如：

  • "回复用中文"
  • "代码使用 Python，遵循 PEP 8"
  • "回答简洁，不要过度解释"

这些指令会被注入到每次对话的 System Prompt 中，等同于你每次对话开头都说一遍这些要求。

08 实际应用建议

窗口管理

  • 一个窗口 = 一个主题。 写代码一个窗口、写文档一个窗口、头脑风暴一个窗口。不要混用。
  • 对话太长时，开新窗口。 把关键结论/代码复制到新窗口的首条消息中，而不是在老窗口里继续。
  • 用 /compact 或 /clear 及时清理。 Claude Code 中，当对话轮数超过 15-20 轮，考虑压缩或重新开始。

输入优化

  • 给必要的上下文，不给多余的。 不要贴整个代码库，只贴相关文件。
  • 把重要信息放在输入的开头或结尾。 减少"迷失在中间"的影响。
  • 用 CLAUDE.md / Instructions 管理重复性指令。 不要每次对话都手动重复相同的要求。

Agent 模式下的注意事项

  • Git 是安全网。 让 Agent 修改代码前确保工作区干净，随时可回退。
  • 先要方案再执行。 让 Agent 先描述它打算怎么做，确认后再动手。
  • 分步走。 复杂任务拆成小步骤，每步完成后 review 再继续。
  • 理解 Agent 的工具。 知道它能做什么（读写文件、跑命令），就能更好地给它指令。

Built for understanding, not for hype.
