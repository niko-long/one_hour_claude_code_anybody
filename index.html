01 Shell 基础

什么是 Shell

Shell 是操作系统提供的命令行界面，用户通过文本命令与系统交互。macOS 默认的 Shell 是 zsh。图形界面能做的事情，Shell 几乎都能做，而且在批量操作、自动化、远程管理等场景下效率更高。

打开终端

macOS 上有两种常用方式：

  • Spotlight 搜索： 按 Cmd + Space，输入 "Terminal"，回车
  • Finder： 应用程序 → 实用工具 → 终端

也可以使用第三方终端应用如 iTerm2，功能更丰富。

常用命令

Terminal — zsh
# 查看当前所在目录
$ pwd
/Users/yourname

# 列出当前目录的文件和文件夹
$ ls
$ ls -la          # 包含隐藏文件和详细信息

# 切换目录
$ cd ~/Desktop     # 进入桌面
$ cd my-project    # 进入子目录
$ cd ..             # 返回上一级
$ cd ~              # 回到用户主目录

# 创建目录和文件
$ mkdir my-project  # 创建文件夹
$ touch index.html  # 创建空文件

# 查看文件内容
$ cat README.md     # 输出整个文件
$ head -20 app.py   # 只看前 20 行

# 删除（谨慎使用）
$ rm file.txt       # 删除文件
$ rm -r folder/     # 删除文件夹及其内容
💡

按 Tab 键可以自动补全路径和文件名。按 ↑ 可以翻出上一条命令。

Hands-on #1

终端基本操作

  1. 打开终端，运行 pwd 查看当前目录
  2. 用 cd ~/Desktop 切换到桌面
  3. 用 mkdir workshop-demo 创建一个文件夹
  4. 用 cd workshop-demo 进入，再用 ls -la 查看

02 Git 与版本控制

为什么需要版本控制

版本控制系统记录文件的每一次变更。在项目开发中，它解决几个核心问题：

  • 回滚： 改坏了可以退回到任意历史版本
  • 协作： 多人同时修改代码，合并变更
  • 追溯： 每一行代码是谁、什么时候、为什么改的
  • 分支： 在不影响主线的情况下开发新功能

Git 是目前最主流的版本控制工具。GitHub、GitLab 等平台提供基于 Git 的远程托管服务。

在 AI 辅助开发中，Git 尤其重要

当 Claude Code 直接修改文件时，Git 是安全网。如果 AI 生成的代码有问题，可以用 git diff 查看改了什么，用 git checkout 撤销。没有版本控制的情况下让 AI 改代码，风险较高。

基本工作流

Terminal — zsh
# 检查是否安装了 Git
$ git --version

# 如果没有，macOS 会提示安装 Xcode Command Line Tools
$ xcode-select --install

# 初始化一个 Git 仓库
$ cd my-project
$ git init

# 查看当前状态
$ git status

# 查看具体改了什么
$ git diff

# 暂存 → 提交
$ git add .
$ git commit -m "add user auth module"

# 查看提交历史
$ git log --oneline

# 撤销未提交的修改
$ git checkout -- .

# 分支操作
$ git branch feature/new-api
$ git checkout feature/new-api
$ git checkout main
$ git merge feature/new-api
📌

建议在让 Claude Code 修改代码之前，先确保工作区是干净的（git status 无未提交变更）。这样如果 AI 改出问题，可以一键回退。

Hands-on #2

初始化一个 Git 仓库

  1. 进入 workshop-demo 目录
  2. 运行 git init
  3. 创建文件：echo "# My Project" > README.md
  4. git add README.md → git commit -m "initial commit"
  5. git log --oneline 查看记录

04 安装 Claude Code

系统要求

  • macOS 13 (Ventura) 及以上
  • 终端（系统自带即可）
  • Git（推荐）
  • Node.js 不是必须的（原生安装器零依赖）

安装

Terminal — zsh
# 原生安装器（推荐，零依赖，自动更新）
$ curl -fsSL https://claude.ai/install.sh | sh

# 或 Homebrew（需手动更新）
$ brew install --cask claude-code

# 验证
$ claude --version

认证

首次运行 claude 时，终端会打开浏览器进入登录页面。完成授权后回到终端即可。凭证保存在 ~/.claude/，后续不需重复登录。

故障排查

Terminal — zsh
$ claude doctor           # 自动诊断
$ source ~/.zshrc         # command not found 时重新加载配置
Hands-on #3

安装并启动 Claude Code

  1. 运行安装命令
  2. claude --version 确认版本号
  3. 运行 claude，完成浏览器认证
  4. 看到欢迎信息即安装成功
⏱️ 5 分钟

05 Claude Code 基础操作

启动位置

Claude Code 启动时会扫描当前目录的文件结构，因此需要先 cd 到项目根目录再启动。

斜杠命令

命令作用场景
/help查看所有命令不确定时
/clear清空对话切换任务
/cost查看 token 消耗成本监控
/model切换模型按任务复杂度选模型
/compact压缩上下文长对话快满时

交互方式

理解代码

"概述一下这个项目的架构"

生成代码

"写一个函数把 JSON 转成 CSV"

修改代码

"把日志模块换成 structlog"

执行 & 调试

"运行测试，分析失败原因并修复"

⚠️

Claude Code 修改文件前会显示 diff 并请求确认。建议始终检查 diff 内容再接受。

Hands-on #4

用 Claude Code 读懂一个项目

  1. 进入项目目录，运行 claude
  2. 输入："这个项目的文件结构是怎样的？"
  3. 输入："如果要添加一个 CSV 读取功能，你会怎么组织代码？先说方案不要直接写。"
  4. 运行 /cost
⏱️ 5 分钟

06 CLAUDE.md：项目级上下文

作用

CLAUDE.md 放在项目根目录，Claude Code 每次启动时自动读取。它的作用类似于给新成员写的 onboarding 文档，告诉 Claude 项目的构建命令、代码规范、架构约定等。

内容建议

重点写 Claude 容易搞错的信息：构建命令、命名规范、技术选型约束、目录结构。

CLAUDE.md
## 项目简介
订单管理 API，FastAPI + PostgreSQL + SQLAlchemy

## 常用命令
- make dev     → 启动开发服务器 (port 8000)
- make test    → pytest
- make lint    → ruff check

## 架构
- src/routes/   → API 路由，按业务模块分文件
- src/models/   → SQLAlchemy 模型
- src/schemas/  → Pydantic schema

## 规范
- 类型注解必须，不使用 Any
- Commit message 用英文祈使句
- 环境变量通过 .env 管理，不硬编码

分层与忽略

Monorepo 中可以在子目录放各自的 CLAUDE.md，按层级加载。.claudeignore 文件（类似 .gitignore）可以排除 node_modules/、.env 等。建议将 CLAUDE.md 提交到 Git。

Hands-on #5

生成 CLAUDE.md

  1. 在 Claude Code 中输入："根据当前项目结构，帮我生成一份 CLAUDE.md"
  2. 检查并修改生成内容
  3. /clear 后重新启动 claude，测试是否生效
⏱️ 5 分钟

07 前后端分离与项目结构

前后端分离

现代 Web 应用通常将前端和后端分开开发和部署。

前端 (Frontend)

用户在浏览器中看到和交互的部分。HTML/CSS/JS，React、Vue 等框架。负责界面渲染。

浏览器端

后端 (Backend)

运行在服务器上，处理业务逻辑、数据存储、权限校验。Python、Node.js、Go 等。

服务器端

API

前后端之间的通信接口。前端通过 HTTP 请求获取数据，后端返回 JSON。

连接层

好处是前后端可以独立开发、独立部署、独立扩展，通过 API 契约协作。

典型项目结构

项目结构示例
my-app/
├── frontend/           # 前端
│   ├── src/
│   │   ├── App.jsx
│   │   └── components/
│   ├── package.json
│   └── vite.config.js
├── backend/            # 后端
│   ├── src/
│   │   ├── main.py
│   │   ├── routes/
│   │   └── models/
│   ├── tests/
│   └── pyproject.toml
├── CLAUDE.md
├── .gitignore
└── README.md
📌

初学者不必一开始就搭建完整的前后端分离架构。可以从一个纯后端 API 或纯前端页面开始，理解基本流程后再组合。

08 Prompt 技巧与项目管理

明确技术栈

在项目开始时明确告诉 Claude 使用的技术栈，避免它自行选择不合适的工具。

对比
# ❌ 模糊
"帮我做一个 Web 应用"

# ✅ 明确
"用 FastAPI 做后端，SQLite 做数据库，
 Pydantic 做数据校验。不需要前端，
 只提供 REST API。Python 3.12。"

让 Claude 反问你

不确定技术方案时，可以让 Claude 扮演顾问角色，通过提问帮你理清需求。这对非专业开发者尤其有效。

Claude Code
"我想做一个内部用的数据看板，展示销售数据。
 我不确定应该用什么技术栈。
 请先问我几个问题来了解需求和限制条件，
 然后再给出方案建议。"

# Claude 可能会问：
# - 数据量大概多大？多久更新一次？
# - 有多少人使用？需要登录吗？
# - 你熟悉哪些编程语言？
# - 部署在内网还是公网？
# - 数据来源是数据库还是 Excel/CSV？

分步执行

避免一次性给出复杂需求。拆分成小步骤，每步确认后再继续。

分步示例
# Step 1: 要方案不要代码
"我要做一个 CSV 转 Markdown 的 CLI 工具。
 先说你打算怎么实现，列出文件结构和步骤。"

# Step 2: 确认后执行
"方案可以，开始第一步：初始化项目结构。"

# Step 3: 逐步推进
"实现核心转换逻辑。"

# Step 4: 补充测试
"为转换函数写 pytest 测试。"

技巧清单

技巧示例
明确技术约束"用 dataclasses 而不是 dict"
要求先规划"先说方案，不要直接写代码"
让 AI 反问"我不确定方案，先问我几个问题"
限定范围"只改 utils.py，不要动其他文件"
要求解释"改完后解释为什么这样改"
迭代修改一个功能完成再做下一个
Hands-on #6

反问式 Prompt 确定方案

  1. 想一个你工作中实际遇到的小需求
  2. 在 Claude Code 中描述需求，并说 "我不确定最佳方案，请先问我几个问题"
  3. 回答 Claude 的提问，评估它给出的方案
⏱️ 5 分钟

10 Agent、Context 与扩展机制

Claude Code 的工作原理

Claude Code 是一个 Agent 循环：接收任务 → 选择工具（读文件、写文件、执行命令）→ 执行 → 观察结果 → 决定下一步 → 循环到完成。它不是一次性生成代码，而是逐步推进，每步基于前一步的结果判断。

Context（上下文窗口）

Claude 的上下文窗口有限。对话过长或项目文件太多时，早期信息可能被截断。管理方式：

  • /compact 压缩对话历史
  • .claudeignore 排除不相关的大文件
  • CLAUDE.md 每次加载会占用上下文空间，保持精简
  • MCP 服务器也消耗上下文，不用的断开

Skills

Markdown 格式的指令文件，定义可复用的知识或工作流。通过斜杠命令调用，或由 Claude 自动加载。适合将重复性任务标准化。

Subagents

独立的子代理，拥有各自的上下文，不继承主会话历史。适合并行处理多个模块或隔离不需要完整上下文的任务。

MCP（Model Context Protocol）

连接外部服务的协议。每个 MCP 服务器暴露一组工具，Claude 在会话中调用。

Terminal — zsh
$ claude mcp install @anthropic-ai/mcp-server-github
$ claude mcp install @anthropic-ai/mcp-server-filesystem
/mcp   # 查看已连接的 MCP 服务器

Hooks

在特定生命周期事件前后执行自定义脚本，配置在 .claude/settings.json 中。

事件触发时机用途
PreToolUse工具执行前拦截危险命令
PostToolUse工具执行后自动格式化
SessionStart会话启动环境检查
SessionEnd会话结束清理、日志

扩展机制对比

机制作用上下文开销
CLAUDE.md持久项目规则，每次自动加载每次加载
Skills可复用知识/工作流，按需加载调用时加载
MCP连接外部服务连接时消耗
Subagents隔离任务，独立上下文不影响主会话
Hooks自动化脚本零（外部脚本）
